# 랭킹 집계 워커 (rank-worker) — 점수 위조(치팅) 근본 차단

랭킹 점수를 **클라이언트가 못 쓰게** 하고, **서버(이 워커)가 "이번 주 완료한 단어 수"를 센다.**
학생이 콘솔에서 `wk:99999` 같은 값을 넣어도 랭킹에 반영되지 않는다.

## 원리 (서버 관측 원장으로 위조 상한 — r6)
- 워커는 로그인 토큰을 검증(OCR 워커와 동일)하고, **학생 본인의 동기화된 완료기록**(`users/{uid}/private/state` 의 `meta.doneByDay`)을 Firestore REST 로 직접 읽어 **"이번 주(월~오늘) 완료한 단어 유니크 수"**를 센다.
- 클라이언트가 보낸 숫자/목록은 신뢰하지 않는다 → 콘솔로 `wk` 를 직접 못 쓴다.
- 배포단어에 국한하지 않고 완료한 **모든 단어(개인등록·기본팩·배포)**를 종류 불문 카운트한다.
- **★ 위조 상한(r6, 핵심)**: `doneByDay`·`words` 는 **둘 다 학생이 자기 문서에 쓰므로** 그 스냅샷만으론 위조를 못 막는다(과거 r5 의 "words 맵 대조" 필터는 자기참조라 무력 — 실제 치팅 사건의 원인). 그래서 워커가 **서버 시계로만 관측하는 백데이트 불가 원장 `att:{uid}`** 로 주간 점수에 **상한**을 씌운다: `점수 = min( 스냅샷 유니크수, Σ_{월~오늘} min(att[그날], RANK_DAILY_CAP) )`(총합은 `RANK_WEEK_CAP` 로 컷). `att` 는 **"서버 기준 오늘" 키에만** 기록돼 과거를 소급 못 심으므로, `doneByDay` 를 통째로 위조해도 **"서버가 실제로 며칠에 걸쳐 관측한 만큼"** 으로 묶인다(원샷 99999 = 즉시 무력화). 정직한 학생은 스냅샷이 상한 이하라 **점수 불변**.
- 점수는 워커만 KV에 기록 → 클라 직접 쓰기가 없다. **선생님 대시보드 '반 랭킹'도 같은 상한을 적용**한다(앱 `weekWordsOf` 가 `/teacher` 로 받은 `att` 로 자기보고 `summary.weekWords` 를 컷). ★워커가 배포돼 `/teacher` 가 응답하면, **`att` 흔적이 아예 없는 학생**(워커에 한 번도 동기화 안 함 = 콘솔로 `summary.weekWords` 만 위조한 치터)은 **0 으로 표시**된다(서버가 안 본 것은 미집계 — 리더보드는 애초에 KV 항목이 없어 목록에서 빠지는 것과 동일 철학). 실제로 앱으로 공부한 학생은 동기화 시 `att` 가 남아 0 이 되지 않는다.

## 배포 (약 5분, Cloudflare 대시보드)
1. **Workers & Pages → Create → Worker** 로 새 워커 생성(예: `wordquest-rank`).
2. 코드에 이 폴더의 `worker.js` 내용을 통째로 붙여넣고 **Deploy**.
   > ⚠️ **"Unexpected token 'export'" 오류가 나면**: 그 워커는 예전 **Service Worker 형식**으로
   > 만들어진 것 → `worker.js` 대신 **`worker-dashboard.js`**(로직 동일한 형식 변환본)를
   > 통째로 붙여넣는다. 기존 바인딩·주소는 그대로 유지된다. 코드를 고칠 땐 두 파일을 함께 갱신.
3. **Settings → Variables and Secrets** 에 추가:
   - `FIREBASE_API_KEY` — `firebase-config.js` 의 `apiKey`(공개값).
   - `PROJECT_ID` — Firebase `projectId` (예: `wordquest-a250d`).
   - `ALLOW_ORIGIN` *(선택)* — 앱 도메인(예: `https://<사용자>.github.io`). 기본 `*`.
   - `RANK_STEP` *(선택)* — 보드 키 갱신 최소 진행 단어수(기본 5). 무료 플랜에서 반이 많아 쓰기가 빠듯하면 10으로.
   - `RANK_FLUSH_MIN` *(선택)* — 잔여 점수 플러시 간격 분(기본 15).
   - `RANK_DAILY_CAP` *(선택)* — **위조 상한의 하루 캡**(기본 60 = 권장 일일분량 20의 3배). 하루에 인정하는 관측 완료수 상한. 정직한 학생이 잘리면 **올리고**, 치팅이 남으면 **내린다**(코드 재배포 불필요, 변수만 바꾸면 즉시 적용).
   - `RANK_WEEK_CAP` *(선택)* — **위조 상한의 주간 캡**(기본 300). 위 일 캡과 함께 주간 점수 상한을 형성.
4. **Settings → Bindings → KV Namespace** 에서 네임스페이스 하나 만들고(예: `wordquest-rank-kv`) 변수명 **`RANK_KV`** 로 바인딩.
5. 배포된 워커 주소를 `firebase-config.js` 의 `window.RANK_ENDPOINT` 에 넣고 커밋·배포.
   ```js
   window.RANK_ENDPOINT = "https://wordquest-rank.<계정>.workers.dev";
   ```
   → 이 값이 채워지는 순간부터 랭킹이 서버 집계로 전환된다(비어 있으면 기존 Firestore 방식으로 폴백하므로, 배포 전까지 앱은 그대로 동작).
6. **배포 확인(토큰 불필요)**: 워커 주소를 브라우저로 그냥 열면 `{"error":"unauthorized","v":"r11"}` 처럼
   나온다 — **`v` 값이 현재 코드 리비전**(worker.js 상단 `REV`)과 같으면 새 코드가 라이브다.
   `v` 필드 자체가 없으면 아직 구버전이 돌고 있는 것.

## 무료 플랜 KV 한도 실사고와 대책 (2026-07-16, r4)
**사고**: Cloudflare KV 무료 플랜은 **쓰기 1,000회/일**(list 도 1,000회/일) 한도가 있는데,
sync 1회당 put 최대 5회 × 전교생 종일 동기화로 한도를 초과 → `put()` throw → `/sync` 전체가
500 → **이후 동기화한 학생들의 보드 키가 안 만들어져 랭킹에 통째로 미등재**됐다.
(선생님 대시보드는 v101부터 Firestore 를 직접 읽어 정상 → "대시보드엔 기록이 있는데 랭킹엔 없다"로 보임)

**r4 대책(코드)**:
- **등재 우선(presence-first)**: 이번 주 보드 키가 **없으면 무조건 기록** — 학생당 주 2회(반+전체)면
  등재는 끝난다. 랭킹에서 학생이 '사라지는' 최악 증상부터 차단.
- **갱신 스로틀**: 등재 후에는 이름/연속 변경, `RANK_STEP`(기본 5)단어 이상 진행, 마지막 기록 후
  `RANK_FLUSH_MIN`(기본 15)분 경과 시에만 갱신 → 학습 중 sync 폭주가 put 폭주로 안 이어진다
  (다른 학생 화면에서 내 점수가 최대 4단어/15분 늦게 보일 수 있는 대신, 쓰기가 하루 한도 안에 안착).
- **한도 내성**: put 실패해도 `/sync` 는 200 + 점수 반환, 응답에 `degraded:true` 를 실어 앱이
  "반영 지연" 안내를 띄운다(조용한 누락 금지).
- **내 행 즉석 병합**: `/board` 가 요청 학생 본인의 점수·연속을 **KV 와 무관하게 즉석 계산**해 목록에
  병합(쓰기 0회) → put 이 밀려도 "내가 랭킹에 안 보인다"는 일은 없다. 전체 보드에서 상위 100 밖이면
  내 행을 끝에 보존해 돌려준다.
- **list 실패 구분**: `/board` 의 KV list 가 한도로 실패하면 빈 목록이 아니라 **503 `kv_limit`** 를
  반환 → 앱이 '빈 랭킹'과 '서버 오류'를 구분해 표시한다.

**운영 참고**: 그래도 무료 플랜 한도는 공유 자원이다. 학생 수가 많아 자주 `degraded` 가 뜨면
① `RANK_STEP=10` 으로 올리거나 ② **Workers Paid(월 $5)** 로 전환하면 KV 한도 문제가 사라진다
(코드는 두 플랜 모두 동일하게 동작).

## (선택) 심층방어 — Firestore 랭킹 쓰기 잠그기
워커로 전환되면 앱은 `leaderboards/*` 를 더는 읽지 않으므로 거기에 위조 값을 넣어도 무해하다.
그래도 깔끔하게 잠그려면, 배포·확인 후 `firestore.rules` 의 두 `leaderboards` 매치의 `allow write` 를 아래로 바꿔 게시한다(읽기도 필요 없으면 함께 `if false`):
```
// leaderboards/_global/entries/{uid} 와 leaderboards/{cid}/entries/{uid} 공통
allow write: if false;   // 점수는 rank-worker(KV)만 기록 — 클라 직접 쓰기 금지
```
`allow delete` 는 남겨두면 과거 위조 항목 정리에 쓸 수 있다.

## 현재 치팅 항목 즉시 제거
- **KV(현행 랭킹 원천)**: Cloudflare 대시보드 → 해당 KV 네임스페이스에서 이번 주 치터 키 삭제 —
  `g:<이번주월요일>:<치터uid>`(전체 보드)·`c:<이번주월요일>:<반ID>:<치터uid>`(반 보드).
  r6 배포 후엔 치터가 다음 `/sync` 만 해도 감소분이 즉시 덮어써지고(자가치유), 그대로 둬도 다음 월요일 새 주 키·TTL 16일로 소멸한다.
- **Firestore(구 폴백 잔재, 있으면)**: `leaderboards/_global/entries/{uid}` · `leaderboards/{반ID}/entries/{uid}` 삭제.

## 선생님 대시보드도 서버 검증 (연속일수·학습일수 위조 차단) — v100
과거엔 선생님 대시보드가 학생이 자기 계정에 쓴 `summary.streak`(연속일수)·`summary.daily`(날짜별 기록)를 **그대로 믿어서**, 학생이 콘솔로 그 값을 바꾸면 대시보드에 그대로 떴다(랭킹만 서버 집계였고 대시보드는 아니었음).
이제 이 워커가 **서버 시계(KST) 기준으로 관측한** 출석·연속일수를 KV에 남기고, 선생님 전용 조회로 내려준다:
- `/sync` 때마다: 서버가 정한 **오늘(KST)** 에 실제 완료가 있으면 `att:{uid}` 에 그날을 기록한다. **날짜 키를 서버가 정하므로** 학생이 과거 날짜/연속을 못 심는다(클라가 보낸 `today` 는 출석 산정에 안 쓴다).
- `mem:{cid}:{uid}` 로 반 명단을 남겨(45일 롤링) 선생님이 학생 출석을 조회할 수 있게 한다.
- **`GET /teacher?class=CLASSID`** — **소유 선생님(또는 마스터)만**. 반 학생별 `{ streak(서버 관측 연속일수), studyDays(학습일수), days(출석맵) }` 반환. 남의 반은 `403`(반 문서 `ownerUid` 를 요청자 토큰으로 확인 → `firestore.rules` 와 동일 신뢰선).
- 앱(`cloud.js teacherBoard`)이 이 값을 받아 대시보드의 🔥연속·📅학습일·히트맵을 **서버 검증값**으로 표시한다. 워커가 구버전이면 `/teacher` 가 `404` → 앱이 자동 폴백하고 대시보드에 "⚠️ 서버 검증 미적용(재배포 필요)" 배너를 띄운다.
- (선택) `MASTER_EMAIL` 변수로 마스터 이메일 재정의 가능(기본 `ranha.park@gmail.com`, `firestore.rules` 와 동일).

> ⚠️ **이 기능은 워커 재배포가 필요하다.** `worker.js` 를 다시 붙여넣고 **Deploy** 하면 즉시 켜진다(새 변수·바인딩 불필요, 기존 `RANK_KV`·`FIREBASE_API_KEY`·`PROJECT_ID` 그대로). 서버 검증값은 **재배포 시점부터** 쌓이므로 배포 직후 며칠은 히트맵이 비어 보일 수 있다(정상).

## 가용성 방어 (DDoS/남용) — r15
공격면: 오픈 가입이라 공격자 풀 = 구글 계정 아무나. Firebase **Spark(무료)** 할당량(읽기 5만·쓰기 2만/일)을 브라우저 루프로 소진시키면 하루짜리 무료 outage(비용 $0·자정 리셋)가 가능하다. r15 는 **코드로 가능한 부분(P1·P2)** 을 적용하고, 근본책(P3·P4)은 콘솔 작업으로 분리한다.

- **P1 — 워커 rate-limit (코드·적용됨)**: `/sync`·`/board` 에 `rlHit`(KV 카운터) uid·분당 상한을 건다. 기본 `RL_SYNC_PER_MIN=10`·`RL_BOARD_PER_MIN=30`(env 로 튜닝, 재배포 불필요). `/sync` 는 요청당 Firestore **2읽기**(doneByDay+classId)라, 한 계정이 루프로 읽기 할당량을 태우던 벡터를 캡한다. rlHit 읽기 실패는 통과(가용성 우선). ★per-uid 상한이라 **버스트**는 막지만 하루 총량은 다계정·저속 반복으로 남을 수 있음 → 근본책은 P3.
- **P2 — verifyFirebase 결과 KV 캐시 (코드·적용됨)**: 매 요청 Identity Toolkit `accounts:lookup` 을 때리던 것을, 토큰 해시(`auth:`+sha256 앞40hex) 키로 `AUTH_CACHE_SEC`(기본 300초) 캐시해 같은 토큰 반복조회의 증폭을 제거한다. 폐기/만료 토큰이 최대 TTL 잔존하나 저위험(수용). ※무효/신규 토큰 플러드는 캐시 미스라 남음 → 더 강한 대안은 ID토큰 **로컬 JWT 검증**(RS256, App Check JWKS 코드와 동일 패턴)으로 네트워크 호출 자체 제거(향후 후보).
- **P3 — Firebase App Check (콘솔·근본책)**: 두 층을 구분한다.
  - **(a) 워커 `/quiz/*`**: 이미 `verifyAppCheck` 게이트 존재(r13). 워커 env `PROJECT_NUMBER` 설정 + `APPCHECK_ENFORCE=true` 로 켠다.
  - **(b) Firestore 직접 쓰기(③b 벡터)**: 학생이 콘솔로 `users/{uid}`·`private/state` 를 무제한 self-write 해 **쓰기 할당량**을 태우는 건 워커로 못 막는다(워커를 안 거침). 이건 **Firebase 콘솔 → App Check 에서 Cloud Firestore 를 enforce** 해야 비앱 클라(콘솔·스크립트) 쓰기가 차단된다. ← ③b 의 실질 근본책. (워커의 `APPCHECK_ENFORCE` env 와는 **별개**다.)
- **P4 — Origin/Referrer 제한 (콘솔)**: 워커 env `ALLOW_ORIGIN` 을 앱 도메인(`https://ranha0924.github.io`)으로 좁히고, Firebase 콘솔에서 웹 apiKey 에 **HTTP referrer 제한**을 건다.

> ⚠️ **적용**: P1·P2 는 **워커 재배포**(r15)만으로 켜진다(새 바인딩 불필요, `RL_*`·`AUTH_CACHE_SEC` 는 선택 env). 확인은 워커 주소를 열어 응답 `v:"r15"`. P3·P4 는 위 **콘솔 작업**이 필요하다.

## 랭킹 위조 재침해 대응 — "App Check 우선(1단계)" (r16)
재침해(랭킹 단어수 위조)의 실제 진입로는 ① 팩 **id-echo**(답=단어 id 그대로 제출 → `packAnswerOk` 통과, id=영어단어이고 단어은행이 공개파일이라 결정적·무지식), ② `/quiz/submit` **무채점 개인단어**(personal 은 정답 대조 없이 무조건 적립 ≤500/주), ③ 옛 att 가 `max(attWk, verWk)` 로 합류. 여기에 App Check 가 꺼져 있어(`APPCHECK_SITE_KEY=""`·워커 `APPCHECK_ENFORCE≠true`) **앱 밖 스크립트가 자기 로그인 토큰만으로 워커를 직접** 때릴 수 있었다.

**정직한 한계**: 단어은행이 오프라인 렌더용으로 전부 공개돼 **모든 문제 정답이 공개적으로 도출 가능**하다 → **인페이지(진짜 페이지의 개발자도구) 위조는 완전차단 불가**. 그래서 사용자 선택으로 **랭킹 리셋·서버-MC 재설계는 2단계로 유보**하고, r16 은 **"App Check 우선"** 만 적용한다:

- **(A) `/sync` 도 App Check 게이트**: 기존 `/quiz/*` 와 동일하게 `verifyAppCheck`. **`APPCHECK_ENFORCE=true` 로 켜기 전까진 도먼트**(무토큰도 통과 = 회귀0). 켜면 **앱 밖 스크립트의 `/sync`(=att·보드 기록) 직접 호출을 차단**. ★`/sync` 는 출석·연속·랭킹을 함께 쓰므로 enforce 시 blast radius 주의 — `docs/appcheck-setup.md` §4-b.
- **(B) 무채점 개인단어 크레딧 제거**: `handleQuizSubmit` 의 personal 분기 미적립(`ver_pers` 미기록), 점수식(`combinedWk`·제출)에서 personal 항·`ver_pers` 읽기 삭제 → 랭킹 = `max(attWk, packWk)`. **가장 무른 곁길(정답 0 으로도 500/주)** 봉쇄.
- **트레이드오프(정직)**: 개인단어는 att(`doneByDay`)로 이미 집계되므로 이 제거는 **att 미포화 학생 표시값 불변**. 단 **att 상한(주 300·일 60) 초과 정직 헤비 개인학습자는 상한 초과분만 감소**(개인은 채점불가라 정직분·위조분 분리 불가). 필요 시 `RANK_WEEK_CAP`/`RANK_DAILY_CAP` 을 올리면 att 관측분을 더 보존한다(단 att-시딩 위조 천장도 함께 상향 — att 는 서버관측·일상한·백데이트불가·App Check 게이트라 무채점 곁길보다 훨씬 어렵다).
- **남는 것(정직)**: 인페이지 팩 id-echo·att 시딩은 **미제거** — App Check enforce(앱 밖만 차단)+상한+이상탐지(교사 ⚠️)로 억제. 근본제거(뜻 4지선다 서버출제·index 채점)는 **2단계**.

> ⚠️ **적용**: **워커 재배포(`v:"r16"`)+앱 배포(v121)** 만으로 **(B) 개인 곁길은 즉시 봉쇄**. **(A) 앱 밖 차단은 사용자 콘솔 작업 필수** — `firebase-config.js` `APPCHECK_SITE_KEY` 기입 + 워커 `APPCHECK_ENFORCE=true`(+`PROJECT_NUMBER`) + 권장 Firestore App Check enforce. **둘(키·enforce)을 반드시 동시에** — 한쪽만이면 `/sync`·`/quiz` 403 로 동기화 전체 중단. 즉시 롤백 = `APPCHECK_ENFORCE=false`. 상세 `docs/appcheck-setup.md`.

## 참고
- 지표 = '이번 주(월~오늘) 완료한 단어 유니크 수'. **개인등록·기본팩·배포 단어 모두 포함**(완료 = 배틀에서 맞혀 `doneByDay` 에 기록된 것). 매주 월요일 새 주 키로 넘어가며 자동 리셋.
- **연속일수(streak) — 랭킹 🔥 = 대시보드 🔥 (r11)**: r10 까지 랭킹 연속은 서버 출석원장(`att`) 걷기값**만** 썼는데, 선생님 대시보드는 `max(att 걷기, 살아있는 자기보고 스칼라, att∪자기보고 히트맵 합집합 걷기)` 를 써서 두 화면 값이 달랐다(대시보드가 큼 = 정방향 불일치, 학생이 sync 를 멈추면 랭킹에 끊긴 연속 잔존 = 역방향 불일치). r11 은 워커가 상태문서에서 자기보고 원천(`meta.streak/lastDay/dailyHistory`, 35일 창·0~99999 클램프)을 함께 읽어 **대시보드와 동일한 `unifiedStreak` 산식**으로 연속을 산정해 보드에 기록한다 → 랭킹 🔥값이 대시보드 🔥값과 같아진다. 또 보드에 감쇠 anchor(`d`)를 남겨 `/board` 가 **오늘/어제가 아니면 0** 으로 표시(끊긴 연속이 랭킹에 남던 문제 제거 · 프리즈 학생도 anchor 확보). r10 레거시 항목은 마지막 기록 시각(`at`)으로 근사 감쇠 후 다음 sync 에 정밀 치유. **트레이드오프**: 랭킹 연속이 자기보고를 포함하게 되어(대시보드가 이미 표시하던 값과 동일 신뢰수준) 콘솔로 `meta.streak` 을 부풀리면 랭킹에도 보일 수 있다(클램프·창으로 상한). **주 순위 지표 `wk` 는 여전히 `att` 상한(위조불가)** — 연속은 동점 tiebreak·표시에만 쓰인다.
- **위조 상한(r6)의 성격**: 원샷·과거소급 위조는 서버 원장(`att`) 상한으로 **막힌다**(원샷 99999 → `RANK_DAILY_CAP` 로 컷, 즉시 1위 불가). 완전한 0% 는 아니다 — **매일 실제로 접속·동기화하며 매일 캡만큼** 위조하면 최대 `RANK_WEEK_CAP` 까지 쌓을 수 있다(클라 권위 앱의 강제 한계). 다만 여러 실제 날에 걸쳐야 하고 값이 상한에 묶이며 `att` 타임스탬프로 **감사 가능**하다.
- **정직한 학생이 상한에 잘리는 일은 드물다**: (a) 요일 간 복습 재완료로 `Σ att[d]` 가 주간 유니크수보다 훨씬 커 헤드룸이 크고, (b) 배틀 종료 시 동기화가 `att` 를 정확히 정착시킨다. 드물게 **복습이 거의 없는 신규 헤비유저**가 소폭 잘릴 수 있으면 `RANK_DAILY_CAP` 을 올린다.
- **대시보드 = 인게임 랭킹 일치는 '동기화 정착 후'**: 둘 다 `min(자기보고/스냅샷, att 상한)` 이나, 대시보드는 학생 문서를 못 읽어 당일분 즉시보정은 리더보드만 한다(방금 푼 오늘분이 대시보드에서 잠깐 낮게 보일 수 있음). ★워커 배포 상태에서 대시보드는 **att 로만 상한**하므로, 그 학생의 이번 주 서버 관측이 아직 없으면(동기화 미착·`/sync` 실패) 표시가 **0 까지 내려갈 수 있다**(치터 차단의 이면 — 서버가 안 본 것은 미집계). 앱으로 실제 공부·동기화하면 즉시 반영된다. 워커 미배포(`vb=null`)면 전원 자기보고값 폴백(무회귀·"⚠️ 서버 검증 미적용" 배너 표시).
- 지표 = '이번 주(월~오늘) 완료 단어(유니크) — 서버 원장 상한 적용'. 상태문서는 mask 없이 읽어 `words`(위조 id 필터)·`doneByDay` 를 함께 받는다. 반 랭킹은 같은 반 학생만, 전체 랭킹은 상위 100.
