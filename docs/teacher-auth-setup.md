# 선생님 권한 보안 셋업 — 이메일 허용목록 (2026-07-15)

## 왜 바꿨나 — 취약점

기존에는 "선생님으로 승격"의 **유일한 방어선이 클라이언트에 하드코딩된 비밀번호**
(`index.html` 의 `const TEACHER_PW='8889'`)였다. 소프트 게이트라서:

1. 소스(또는 브라우저 개발자도구)를 열면 비밀번호가 그대로 보이고,
2. 비밀번호를 몰라도 콘솔에서 `Cloud.chooseRole('teacher')` 한 줄이면 승격됐다.
   (Firestore 규칙이 `role` 값이 문자열이기만 하면 통과시켜, 서버 통제가 없었음.)

실제로 한 학생이 이 방식으로 스스로 선생님이 되었다.

## 어떻게 고쳤나 — 서버 허용목록

선생님 자격을 **서버(Firestore 규칙)의 허용목록으로만** 인정한다. 워커도, 비밀번호도,
서비스계정도 필요 없다.

- `firestore.rules`: 선생님이 될 수 있는 계정을 **`teacherAllow/{이메일}` 컬렉션에 있는
  계정**으로 한정. `role='teacher'` 쓰기·반 생성/수정/코드·`classPacks` 쓰기를
  `isTeacher()`(=허용목록에 있음) 또는 마스터만 허용.
- `teacherAllow` 에 쓰기는 **마스터만**(규칙). 학생은 자기를 못 넣으므로 **셀프 승격 불가**.
  → 콘솔에서 `Cloud.chooseRole('teacher')` 를 직접 호출해도 서버가 거부(`permission-denied`).
- 클라이언트: 비밀번호 게이트 삭제. "선생님으로 시작"은 그냥 서버에 시도하고,
  허용목록에 없으면 "선생님 권한이 없어요" 안내.

```
로그인 계정 ──role='teacher' 쓰기 시도──▶ firestore.rules
                                          teacherAllow/{내 이메일} 있나?
                                          ├ 있음 → 허용(선생님)
                                          └ 없음 → 거부(학생 그대로)
teacherAllow 에 쓰기는 마스터만 → 학생이 스스로 못 넣음 → 우회 불가
```

## 나중에 할 일 — 선생님 추가 (이메일 알게 됐을 때, 콘솔에서)

코드 수정·재배포 **전혀 없음**. Firebase 콘솔에서 문서만 추가하면 끝:

1. Firestore Database → **컬렉션 시작**(또는 기존) `teacherAllow`
2. **문서 ID = 그 선생님의 Google 이메일**(로그인에 쓰는 그대로, 보통 소문자.
   예: `teacher@gmail.com`). 필드는 아무거나(예: `name: "김선생"`) 또는 비워도 됨.
3. 저장 → 그 계정으로 로그인하면 즉시 선생님. (이미 로그인 중이면 앱 새로고침.)
   - 마스터 `ranha.park@gmail.com` 는 목록에 없어도 **항상 선생님**(규칙상 특별계정).
   - 선생님 해임 = 그 문서 삭제.
   - ⚠️ **재성 이메일은 넣지 말 것**.

> 이메일이 토큰과 정확히 일치해야 한다(대소문자 포함). 잘 모르면 콘솔
> **Authentication** 탭에서 그 계정의 이메일을 그대로 복사해 문서 ID 로 쓴다.

## 참고

- 이 방식은 **공유 비밀번호가 없다** → 유출 걱정 없음. 선생님 명단은 마스터가
  콘솔에서 문서로 관리(추가/삭제).
- 더 자동화하고 싶으면(선생님이 자주 바뀜) 이메일 대신 uid 기반이나 관리 UI 를
  붙일 수 있다. 현재 규모(반 몇 개)에는 콘솔 문서 관리가 가장 단순하고 안전.
- `sw.js` `VERSION` / `?v=` 는 v70 으로 올려 옛 `index.html`(8889 포함) 캐시를 무효화.
