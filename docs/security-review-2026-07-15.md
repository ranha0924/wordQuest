# WordQuest 구조·보안 리뷰 (2026-07-15)

> 대상: 라이브 중인 전체 시스템 — index.html(로컬 저장·렌더링), cloud.js(동기화), firestore.rules, ocr-worker/worker.js, sw.js, 배포 워크플로.
> 관점: (1) 구조적 취약점(학생 학습 데이터 유실) (2) 보안 취약점.
> 검수: Claude 감사 에이전트 통과 92/100 (기준 90). 코드 수정은 별도 세션(Opus)에서 진행 예정.

## 종합 평가

전반적으로 잘 방어된 설계다. index.html은 모든 신뢰불가 입력(랭킹 이름, 학생 이름/이메일, 반 이름, 배포 단어, OCR/붙여넣기 단어)을 `esc()`로 일관 이스케이프하고, 학생이 위조 가능한 숫자 필드는 정수 강제변환한다. firestore.rules는 "본인만 쓰기 + 타입 검증 + 코드 열거 차단" 구조가 견고하다. OCR 워커는 토큰 검증 + 서버측 한도로 API 키와 비용을 보호한다.

**가장 큰 리스크는 보안이 아니라 "로컬 저장 실패의 침묵"과 "원격 백업 부재"다.** 학생 데이터 보호가 목적이라면 이 둘부터 고쳐야 한다.

## (1) 구조적 취약점 — 학습 데이터 유실 경로 (우선순위순)

### 1. [높음] persist() 저장 실패가 조용히 삼켜짐 — `index.html:1204-1213`
`localStorage.setItem` 실패(quota 초과, Safari 프라이빗 모드, 카카오톡 인앱 웹뷰 등)를 `catch(e){}`로 무시한다. 비로그인 학생은 몇 주치 공부가 메모리에만 있다가 탭을 닫는 순간 전부 사라지며, 아무 경고도 없다. persist() 호출부 17곳 전부가 이 침묵 실패에 의존.
→ **개선**: 실패 시 1회 배너 표시 + "지금 백업 내보내기 / 로그인" 유도.

### 2. [높음] 손상된 저장 JSON → 샘플 데이터로 덮어씀 — `index.html:1091, 1154-1160`
저장 blob이 손상되면(이전 quota 이벤트로 잘린 쓰기, 크래시 등) `saved=null` → 첫 방문으로 간주 → 샘플 20개 시딩 후 즉시 `persist()`가 원본을 덮어쓴다. 비로그인 사용자는 복구 불가. (로그인 사용자는 `sample:true` 태그 덕분에 클라우드 병합이 샘플을 버려서 완화됨 — cloud.js:131-139.)
→ **개선**: 샘플 시딩 전에 원본 blob을 보조 키(`wordquest_v1_corrupt`)에 보관.

### 3. [중간] applyMerged가 빈/축소 병합도 즉시 로컬 반영 + 저장 — `index.html:2974-2991`
`m.words=[]`도 `Array.isArray` 검사를 통과해 로컬 단어 전체를 지우고 바로 persist한다. cloud.js 병합 버그 하나가 곧 로컬 파괴로 직결되는 구조(안전벨트 없음).
→ **개선**: 축소 가드 — 병합 결과가 기존 대비 대폭 작으면(예: 50% 미만) 적용 보류 + 경고.

### 4. [중간] 원격 백업 부재 + 트랜잭션 없는 whole-doc 쓰기 — `cloud.js:199`
Firestore `private/state`가 유일한 원격 사본인데 read→merge→`setDoc`(문서 전체 교체)가 트랜잭션 없이 수행된다. 병합 버그·악성 클라이언트·실수 한 번이면 영구 파괴이고, PITR/정기 export가 없어 복구 수단이 없다.
→ **개선**: 마스터 계정 기반 주기적 전체 export 스크립트(무료 플랜에서 가능) 또는 Blaze 전환 + PITR.

### 5. [중간] 클라이언트 시계 의존 LWW — `cloud.js merge()`
단어별 병합 기준이 `updatedAt=Date.now()`(로컬 시계). 시계가 미래로 틀어진 기기가 병합을 영구 지배해, 다른 기기의 정상 진도가 계속 진다. streak/랭킹 날짜(`todayStr`)도 로컬 시계 기준이라 조작·오작동에 취약.

### 6. [중간] 1MB 문서 한도 + 동기화 실패 원인 미구분 — `cloud.js:203-206`
`private/state`가 단일 문서(현재 ~130KB/648단어 추정). 팩 추가·OCR로 수천 단어가 되면 `setDoc`이 실패하는데, catch가 모든 실패를 "동기화 보류(오프라인?)"로만 표시해 **영구 동기화 중단을 사용자가 알 수 없다**.
→ **개선**: 실패 원인(권한/크기/네트워크) 구분 표시. 장기적으로 단어 분할 저장 고려.

### 7. [낮음-중간] SW가 비정상 응답을 오프라인 셸로 캐시 — `sw.js:62`
`handleNavigate`가 모든 내비게이션 응답을 `./index.html` 키로 무조건 캐시 — 404/500 페이지나 vocamon.html 응답이 오프라인 셸을 오염시켜 오프라인 부팅이 깨질 수 있다(데이터는 무사하나 백업 내보내기 접근 불가).
→ **개선**: `res.ok` + 경로 확인 한 줄.

### 8. [낮음] 기타
- **가져오기 검증 부실 + migrateWords 무가드** (`index.html:2091-2111, 1198`): `words:[null,...]` 형태의 백업 파일이 이후 부팅 브릭(스크립트 전체 사망)을 유발할 수 있음. 요소 단위 검증 필요.
- **리셋 버튼**: confirm 1회로 로컬+원격 동시 삭제 (`index.html:2112-2118`). 삭제 전 자동 백업 다운로드가 거의 공짜 보험.
- **단어 id = 소문자 철자**: 기기 A `book-책`, 기기 B `book-예약하다` 등록 시 병합에서 한쪽 의미·진도가 조용히 소실.

## (2) 보안 취약점 (우선순위순)

### 1. [양호] XSS — 방어 잘 됨
index.html의 innerHTML 싱크 61곳 확인 — 신뢰불가 값은 전부 `esc()` 통과, 숫자는 `int()`/`|0` 강제변환. `eval`/`document.write`/`postMessage` 없음. 예외:
- **vocamon.html:448** (레거시 프로토타입): 백업 가져오기 경유 self-XSS 1건 — 어차피 정리 대상 파일이므로 **삭제 권장**.
- **CSP 메타 태그 부재** — 심층방어로 추가 권장.

### 2. [중간] 랭킹 `days` 맵 무검증 — `firestore.rules` · ✅ 해소(v75)
name/wk/streak만 검증하고 `days` 값은 문자열·거대 맵도 허용, 추가 필드도 무제한. `getRank`/`getGlobalRank`가 days로 주간 합을 재계산하므로 문자열 주입 시 **정렬 순서가 오염**되고(NaN 비교) 문서당 1MB까지 정크 저장 가능.
→ **적용(v75)**: 규칙에 필드 화이트리스트(`hasOnly`) + `days is map`·`days.size()<=9` 추가. 값 자체는 rules로 순회 검증 불가하므로, `cloud.js`의 `weekSum`이 하루 500·주간 2000 상한으로 클램프하여 조작·NaN 이 집계/정렬에 반영되지 않게 방어(심층방어).

### 3. [중간] 열린 가입 + 전체 랭킹 노출 (양방향)
구글 계정만 있으면 누구나 로그인 가능해서:
- **쓰기**: 전체 랭킹에 임의 이름(40자) 게시 가능 — 학생 전원에게 노출되는 부적절 닉네임/스팸 벡터(XSS는 아님). 마스터 삭제 권한은 사후 대응뿐.
- **읽기**: `_global` 랭킹 항목(실명 + 최근 8일 학습 활동 + streak)을 **외부인 누구나** 읽을 수 있음 — 미성년자 실명 노출.
→ **개선**: 반 소속 사용자만 전체 랭킹 참여(읽기·쓰기 모두), 또는 표시명 필터링.

### 4. [중간] 점수 위조(치팅) — 진행중(구조 검증 완료 / 근본 차단 설계)
랭킹 수치는 클라이언트가 자유 기록(타입만 검증) — 콘솔에서 `wk`/`days` 위조로 1등 등극 가능했음(실제 사건 발생).
→ **하지 않은 것(의도적)**: 값 상한(cap). v75에서 하루 500·주간 2000 상한을 넣었으나, **진짜 열심히 해서 높은 값을 낸 학생까지 깎는 부작용**이 있어 v76에서 제거. 상한은 근본 해결도 아님(상한 안에서 여전히 조작 가능).
→ **적용(v76)**: `firestore.rules`·`cloud.js` 모두 값 상한 없이 **구조 검증만**(필드 화이트리스트·타입·음수 방지·`days` 맵 ≤9키) + 숫자 타입가드(문자열/NaN 주입에 의한 정렬 오염 차단).
→ **근본 차단 설계(권장·미구현)**: "완료 단어수"를 **서버 집계**로 전환 — OCR 워커가 이미 쓰는 패턴(Cloudflare 워커가 Firebase 토큰 검증 후 KV에 uid별 카운터, 기기·브라우저·콘솔로 우회 불가)을 랭킹에 재사용. `leaderboards/*` 클라이언트 쓰기는 규칙에서 전면 거부하고, 워커만 기록. 클라가 완료한 **배포 단어 id**를 보고 → 워커가 토큰검증·중복제거(주간)·배포단어 검증 후 증가 → 상한이 아니라 **"배정된 단어를 전부 했을 때"가 자연 천장**이 되어 99999 류 조작이 원천 불가. 지표는 '이번 주 완료한 배포 단어 수'로 소폭 변경.
→ **현행 조작 항목 제거**: 소유 선생님/마스터가 `leaderboards/{cid|_global}/entries/{uid}` 삭제(콘솔 또는 규칙 delete 권한).

### 5. [낮음] 제외(removed) 학생이 반 랭킹에 계속 등장
`removed`는 대시보드 필터일 뿐 — `users.classId`가 남아 있어 랭킹 쓰기 규칙(`requesterClassId()==cid`)을 계속 통과하고, `getRank`도 removed를 필터하지 않음.

### 6. [낮음] classCodes 스쿼팅 — `firestore.rules:99-100`
로그인 사용자는 임의 코드→자기 반 매핑을 선점 생성 가능(생성 전용이라 기존 코드 탈취는 불가). 실질 피해 낮음.

### 7. [낮음] OCR 워커 — `ocr-worker/worker.js`
- `ALLOW_ORIGIN` 기본 `*` — 배포 도메인으로 제한 권장.
- 전체 500/day 캡을 외부 구글 계정들이 소진 가능(기능 DoS; 비용은 Anthropic 지출 한도로 방어됨).
- KV 카운터 race는 근사치로 설계상 인지된 트레이드오프 — 적절. upstream 오류 detail 300자 노출은 무해 수준.

### 8. [낮음] 운영 보안
- **교사 모드 비밀번호 `TEACHER_PW='8889'` 하드코딩** (`index.html:3005`): 배포 소스에서 누구나 읽을 수 있는 장식용 게이트일 뿐이며, rules상 반 생성은 어차피 로그인 사용자 누구나 가능. 진짜 접근 통제는 소유권 기반이라 피해는 제한적이나, 평문 비밀번호가 소스에 있다는 점은 인지 필요.
- **마스터 이메일 1개**가 전 학생 데이터 열람권 — 해당 구글 계정 2FA 필수.
- Firebase apiKey **HTTP 리퍼러 제한** 미적용이면 적용 권장.
- classes update 규칙이 code/name/ownerUid 외 잡필드 추가를 허용 (`firestore.rules:84-89`) — 경미.
